# Field Notes 직접 수정 안내

이 사이트는 GitHub Pages와 GitHub Actions로 자동 게시됩니다. Jekyll, Ruby, Node.js를 컴퓨터에 설치하지 않아도 GitHub 웹사이트만으로 글을 고치고 추가할 수 있습니다.

## 가장 쉬운 수정 방법

1. https://github.com/JUNandCHU/JUNandCHU.github.io 에 로그인합니다.
2. 수정하려는 파일을 엽니다. 글은 content/posts.ts, 첫 화면 문구는 content/site.ts에 있습니다.
3. 파일 오른쪽 위의 연필 아이콘(Edit this file)을 누릅니다.
4. 내용을 수정합니다. 따옴표, 쉼표, 대괄호와 중괄호는 그대로 유지하는 것이 중요합니다.
5. 오른쪽 위의 Commit changes 버튼을 누릅니다.
6. 짧은 변경 설명을 입력하고 Commit directly to the main branch를 선택한 채 다시 Commit changes를 누릅니다.
7. 저장소 위쪽의 Actions 탭에서 Publish to GitHub Pages 작업이 초록색 체크 표시가 될 때까지 기다립니다. 보통 몇 분 안에 반영됩니다.
8. https://junandchu.github.io/ 를 새로고침합니다. 이전 화면이 보이면 강력 새로고침을 하거나 1~2분 뒤 다시 확인합니다.

## 어떤 파일을 고치면 되나요?

- content/posts.ts: 글 목록, 본문, 날짜, 태그, About 내용
- content/site.ts: 상단 소개, 현재 관심사, Now 카드, 인용문, 푸터 문구
- app/layout.tsx: 브라우저 탭의 제목과 검색 결과용 설명
- app/globals.css: 색상, 글꼴, 간격, 모바일 화면 등 전체 디자인
- app/page.tsx: 화면 구조와 검색·분류·글 열기 같은 동작

대부분의 일상적인 운영은 content 폴더의 두 파일만 수정하면 됩니다.

## 기존 글 수정하기

content/posts.ts에서 제목을 검색한 뒤 해당 글의 title, dek, body, tags, date 항목만 고칩니다.

- title: 글 제목
- dek: 목록에 보이는 한두 문장의 소개
- body: 본문 문단 목록. 각 문단은 따옴표로 감싸고 문단 사이에 쉼표를 둡니다.
- tags: 검색과 분류에 쓰는 태그 목록
- date: 화면에 보여 줄 날짜
- read: 예상 읽기 시간
- category: Research, Fiction, Marginalia 중 하나

## 새 글 추가하기

content/posts.ts의 entries 목록 안에서 기존 글 하나를 통째로 복사해 붙여 넣은 뒤 아래 항목을 바꿉니다.

1. id를 다른 글과 겹치지 않는 새 숫자로 바꿉니다.
2. category를 Research, Fiction, Marginalia 중 하나로 정합니다.
3. date, read, mark, title, dek를 바꿉니다.
4. body의 각 문단을 따옴표로 감싸 입력하고 문단 사이에 쉼표를 둡니다.
5. tags를 필요한 키워드로 바꿉니다.
6. 복사한 글 블록의 마지막 중괄호 뒤에 쉼표가 필요한지 앞뒤 글을 보고 확인합니다.
7. Commit changes로 저장한 뒤 Actions 결과를 확인합니다.

mark는 R—03, F—02, M—03처럼 분류별 순번을 알아보기 쉽게 적는 표시일 뿐입니다.

## 공개 전에 꼭 확인할 것

연구 글은 일반적인 과학 개념과 이미 공개된 사실, 연구 과정에서 얻은 보편적인 생각만 적는 편이 안전합니다. 다음 정보는 넣지 마세요.

- 실명, 소속 학교·회사·연구실, 건물이나 도시 등 위치 정보
- 동료, 공동 연구자, 의뢰인, 협력 기관을 알아볼 수 있는 정보
- 공개되지 않은 물질명, 시료명, 조성, 농도, 공정, 장비 조건
- 미공개 데이터, 수치, 결과, 가설, 실패 원인, 다음 실험 계획
- 내부 프로젝트명, 파일명, 과제·계약·특허·심사 관련 정보
- 사진이나 화면 캡처에 보이는 라벨, 이름표, 문서, 위치 정보

애매하다면 구체적인 대상과 수치를 지우고 관찰 방법, 측정의 한계, 질문을 고르는 방식처럼 일반적인 수준으로 다시 쓰세요. 공개 여부가 확실하지 않은 내용은 게시하지 않는 것이 좋습니다.

## 게시가 실패했을 때

1. 저장소의 Actions 탭을 엽니다.
2. 빨간색으로 실패한 Publish to GitHub Pages 실행을 누릅니다.
3. build 단계의 오류 메시지에서 파일명과 줄 번호를 확인합니다.
4. 대개 빠진 따옴표, 쉼표, 대괄호 또는 중괄호가 원인입니다. 직전에 수정한 파일을 다시 열어 기존 글의 구조와 비교합니다.
5. 고쳐서 다시 Commit하면 자동으로 새 게시가 시작됩니다.

GitHub의 파일 History에서 이전 버전을 열어 내용을 확인할 수 있으므로, 실수하더라도 다시 고칠 수 있습니다. 다만 민감한 내용을 한 번 Commit하면 삭제해도 저장소 기록에 남을 수 있으니, 기밀 정보는 처음부터 입력하지 마세요.
